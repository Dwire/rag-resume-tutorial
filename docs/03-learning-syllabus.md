# Learning Syllabus

The numbered implementation steps. **One step per session** unless the student explicitly asks to continue. Each step follows the teaching contract: planning explanation → go-ahead → build → checkpoint review with interview prep + quiz → update `LEARNING_LOG.md`.

**Testing rule:** from Step 1 onward, every step's build includes lightweight `pytest` tests for its deterministic parts (chunker boundaries, retriever counts, word-count validators). LLM stages get *output validators and evals* instead of unit tests — knowing that distinction is itself an interview answer.

| # | Step | What gets built | Concepts learned | Interview topics it maps to |
|---|---|---|---|---|
| 0 | **Foundations & planning** ✅ | CLAUDE.md, LEARNING_LOG.md, these three docs, initial ADRs. No code. | The RAG pipeline end-to-end at a conceptual level; framework vs. hand-rolled | "Walk me through a RAG architecture"; "why did you make the design choices you made?" |
| 1 | **Project setup & document loading** | `venv`, `requirements.txt`, folder layout (`src/`, `tests/`, `personal_docs/`, `output/`), `.env` + `.gitignore`, git init, hello-world script proving the Anthropic SDK connects; `src/loader.py` walking `personal_docs/` with metadata (filename, headings); 2–3 real seed documents; first pytest tests | Dependency isolation, secrets management; ingestion, document representation, why metadata matters downstream; testing a pipeline stage | Engineering hygiene; API-key security; "how did you ingest and structure source data?" |
| 2 | **Chunking strategies** | `src/chunker.py`: hand-written paragraph/heading-aware splitter with size cap + overlap; tests for boundary cases; a script to eyeball chunks at different settings | Chunk size vs. retrieval precision, overlap, semantic boundaries, tokens vs. characters | **The** classic RAG question: "how did you chunk and why? what happens if chunks are too big/small?" |
| 3 | **Embeddings & similarity by hand** | `src/embedder.py` with sentence-transformers; a "similarity playground" script: embed sentences, compute cosine similarity manually with NumPy, see meaning-as-geometry | What an embedding is, vector dimensions, cosine similarity (computed by hand once!), semantic vs. keyword matching | "What are embeddings? How does similarity search work mathematically?" |
| 4 | **Hand-rolled vector store** | `src/vector_store.py`: a small `VectorStore` interface (`add`, `search`) + a NumPy brute-force implementation; wire load→chunk→embed→store into one `index` command | What a vector DB actually does under the hood (linear scan + cosine), programming to an interface so implementations can swap, cost of brute force | "What does a vector database do?" — answered from having built one; interface-driven design |
| 5 | **Retrieval & multi-query search** | `src/retriever.py`: parse a job posting into **multiple targeted queries**, top-k search per query, merge/dedupe/rank; a `retrieve` command showing chunks + scores | Query construction, top-k, score thresholds, why multi-query beats embedding the whole posting as one blob, inspecting retrieval quality | "How do you turn a user request into good retrieval queries? How do you know retrieval is working?" |
| 6 | **Swap in ChromaDB & compare** | `ChromaVectorStore` implementing the same interface; persistence to `./chroma_db/`; side-by-side comparison with the NumPy store (results, speed, ergonomics) | What a real vector DB buys you (persistence, metadata filtering, upserts) vs. what it doesn't (magic); when you'd need ANN indexes (HNSW) at scale | "What vector store did you use and why? What changes at production scale?" — with an I-built-both answer |
| 7 | **Prompt engineering for grounded generation** | `src/prompts.py`: the grounding system prompt (use ONLY provided excerpts, no fabrication), context injection format, constraint wording; test on toy inputs | Prompt structure (system vs. user), grounding/hallucination prevention, instruction hierarchies, context-window budgeting | "How did you prevent hallucination?" — the #2 RAG interview question |
| 8 | **Resume generation** | `src/generator.py` (resume path): Claude API call wiring retrieval → prompt → output; single-page **structure** constraint (sections, bullet density); save to `output/` | LLM API mechanics (messages, system prompts, max_tokens), enforcing structural constraints, post-generation validation | "How do you enforce output structure? What API parameters matter?" |
| 9 | **Cover letter generation** | Cover-letter path in the generator: ≤300-word **length** constraint with a word-count validator (retry/trim loop), narrative tailoring to the posting | Length vs. structure constraints, validate-and-retry patterns, tone/narrative control via prompt | "What do you do when the model ignores a constraint?" |
| 10 | **Evaluation & iteration** | A small eval harness: golden queries ("does a DevOps posting retrieve the K8s project?"), precision/recall@k informally, groundedness spot-checks; experiments with chunk size / k; record findings | Retrieval metrics, groundedness checking, systematic tuning; hybrid search (BM25 + vectors) & rerankers discussed as next steps | "How do you evaluate a RAG system?" — the question that separates tutorial-followers from engineers |
| 11 | **Simple UI** | `app.py` in Streamlit: paste a posting, click generate, see retrieved chunks (transparency!) and the outputs side by side | Separation of pipeline and presentation; showing retrieval provenance in a UI | Demo material; "how would users trust the output?" |
| 12 | **Wrap-up: mock interview** | No new code. Full mock interview: architecture walkthrough, deep-dives per stage, trade-off grilling, "what would you change at scale." Produce a 1-page study sheet from the ADRs + log. | Synthesis and retrieval (the human kind) | Everything above, under pressure |

**Ordering logic:** steps 1→9 build the pipeline left to right through the architecture diagram. The vector store is deliberately built by hand (4), *used* for real retrieval (5), and only then replaced by ChromaDB (6) — so the comparison is felt, not read about. Generation is split into resume (structure constraints) and cover letter (length constraints) because the interesting problems differ. Evaluation comes once there's something to evaluate; UI comes last because it teaches no RAG.

**Flex points:** URL fetching for job postings (instead of pasted text) can be added around Step 5 or skipped; PDF loading can be added to the loader if ever needed.
